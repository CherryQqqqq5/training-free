# Progress Snapshot (2026-04-21)

This document records the current verified progress of the training-free self-evolution workstream.

Scope:

- Only includes changes and experiment results that were explicitly verified in local code or remote runs.
- Does not treat unverified hypotheses as completed work.
- Does not mix compatibility-layer gains with compiler/self-evolution gains.

## 1. Current Position

The repository now has a reliable Phase-1 experimental harness for BFCL, but it is not yet a self-improving system that can consistently raise BFCL scores through compiled patches.

What is already real:

- BFCL-first protocol is pinned and documented.
- Baseline / candidate artifact contracts are now explicit.
- Compile outputs are no longer treated as valid candidates by default.
- Baseline / candidate runs now record enough provenance to compare runs safely.
- BFCL online execution is gated by a preflight check so obviously broken runs do not burn time or tokens.
- Compatibility-layer fixes now make the BFCL pipeline trustworthy enough to analyze real model behavior.
- Early Phase-2 targeted experiments can already produce measurable target-slice uplift, even though the architecture is still transitional.

What is still missing:

- A fully clean policy architecture across compiler, runtime, and selector; the repository is still in a compatibility-preserving transition.
- Broader compiler templates that target high-value no-tool / wrong-stop failures beyond the currently upgraded no-tool families.
- A real outer-loop over historical candidate patches.
- A learning signal that feeds selector outcomes back into proposal generation.

Accurate architecture boundary:

- The repository already behaves like a credible experiment operating system with a closed-loop harness.
- The current Phase-2 results appeared before a clean policy-compiler architecture was fully realized.
- A first explicit `DecisionPolicySpec` now exists in IR and is emitted for the current high-value global no-tool rules, while runtime still overlays it onto legacy verification/trigger fields for compatibility.
- Compiler/runtime behavior therefore contains real Phase-2 semantics in a transitional structure, rather than a finished self-evolving policy layer.

## 5. Current Diagnosis

At this point, benchmark corruption is no longer the primary explanation for poor harder-slice performance.

The strongest evidence is:

- `simple_python` runs normally and scores high enough to prove the online chain is healthy
- `multi_turn_miss_param` now produces complete artifacts instead of malformed or incomplete result bundles
- dominant failures in harder slices are semantic / behavioral (`empty_tool_call`, `clarification_request`), not transport / layout failures

Current problem statement:

The system can now reliably observe, classify, and archive failures, but it still does not reliably transform high-value behavioral failures into request-time decision improvements.

## 6. What the Current System Is Good At

The current system is already good at:

- running official BFCL through an external evaluator plus local proxy
- keeping baseline and candidate provenance separate and comparable
- making BFCL result artifacts machine-checkable
- mining and classifying runtime failures
- compiling defensive rule forms:
  - prompt fragments
  - tool guards
  - argument sanitizers
  - verification contracts
  - fallback routing

This is enough to claim:

- the harness is real
- the compiler scaffold is real
- the candidate lifecycle is real
- the selector contract is real

## 7. What the Current System Is Not Yet Good At

The current system is not yet good at:

- representing "what tool should come next"
- representing "when the agent must continue instead of stopping"
- representing "when completion is allowed only after tool evidence"
- compiling `empty_tool_call` / `hallucinated_completion` / `natural_language_termination` into request-local policy rules
- reusing prior accepted / rejected patches as search memory
- learning from selector outcomes

This is why the system currently behaves more like:

- a reliable experiment operating system

than:

- a true self-evolving policy learner

## 8. Boundaries and Cautions

The following statements should not be made based on current evidence:

- "compiler-generated patches already produce stable BFCL gains"
- "the current runtime policy layer is a performance enhancer"
- "outer-loop self-evolution is already implemented"

The following statements are supported:

- Phase-1 harness and artifact contract are now substantially more trustworthy
- compatibility-layer fixes are enough to run and analyze official BFCL subsets
- the dominant remaining bottlenecks are decision-policy failures, not benchmark breakage

## 9. Immediate Next Step

The next major step should not be another compatibility patch.

It should be a Phase-2 design that upgrades the system from:

- failure observation + defensive patching

to:

- decision-policy synthesis + historical patch search + scoped request-time control

That design is written separately in:

- [docs/phase2_design.md](/Users/cherry/.codex/worktrees/3253/training-free/docs/phase2_design.md)
